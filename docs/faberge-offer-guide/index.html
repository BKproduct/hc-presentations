AAS Process Guide — Internal

From Zero to Polished Deliverable

Lessons from the Faberge TrendWatch build: how to reach final-quality output from the first iteration, not the fifth.

AI Automation Studio • May 2026 • For Sima + AAS delivery team
Context

What we built and why this guide exists

We built a weekly demand-intelligence system for Sergey Mosunov, CEO of Faberge. The system collects signals from 12 sources, processes them through an embedding-and-scoring pipeline, and delivers a structured weekly brief to his marketing, sales, and design teams.

The build took multiple iteration cycles, each one fixing something that should have been right the first time. This guide captures what caused each rework, and gives you the checklist to avoid it.

The artefacts: live pipeline code (Phase 1 running), a commercial PDF offer in AAS design, and a published system overview at hc-presentations.

Iteration Forensics

What required rework and why

Every item below caused at least one full rewrite pass. Most were preventable with a tighter client brief upfront.

1
Sources

Amazon and Etsy were in the initial source list

Client asked: "Why do we need Amazon?"

Faberge average selling price is £5,000-£500,000. Amazon and Etsy serve the mass market. Including them signals you have not done segment homework. Remove any source whose median buyer is outside the client's ASP range before presenting the architecture.

2
Sources

Pinterest role was misframed as brand presence

Client asked: "How active are reference brands on Pinterest?"

Luxury brands have weak official Pinterest presence. The real value is bridal aesthetic search velocity: brides save board ideas 3-9 months before purchase. If you present Pinterest as a brand monitoring tool, the client will correctly dismiss it. Lead with the buyer behaviour use case, not the brand presence angle.

3
Geography

Russia was included in the geography list

Removed silently after client review.

For any client with international ambitions in 2024+, Russia is a politically sensitive inclusion. Do not put it in default geo lists. Only add if the client explicitly requests it and the project context is clearly domestic/non-sanctioned.

4
Competitors

Graff was in the competitor peer set

Removed after client review.

Always confirm the competitor peer set with the client before building it into config. Do not assume. A two-line message ("Here's the 9-brand peer set I'm using for comparison, please confirm or adjust") prevents a config and PDF rewrite.

5
Commercial offer

Currency was USD in a GBP-context proposal

Faberge is a UK-headquartered luxury brand. Client is based in London.

Always check client HQ and invoice jurisdiction before setting currency. Default is USD from most tools; always override to match client billing context. UK client = GBP. EU client = EUR. Never let tooling defaults slip through to a client document.

6
Commercial offer

Timeline was too optimistic by 2x

Client asked to double all phase durations.

Initial estimate was 2 weeks per phase. Client-facing estimates should include buffer for feedback cycles, API credential delays, and integration surprises. Rule of thumb: take your technical estimate, add 50% for client review cycles, add 25% for credential/access onboarding. For a 4-phase intelligence system, 8 weeks is the minimum credible commitment.

7
Commercial offer

Cost structure mixed infrastructure and AAS work without separation

Client wanted to understand what each line was paying for.

Separate three distinct cost buckets in every proposal: (1) direct API/token spend the client could theoretically pay themselves, (2) AAS stewardship - the value of maintained connectors and model upgrades, (3) upgrade reserve - buffer for new model generations. Bundling these hides the value of ongoing work and invites the client to try to self-manage it.

8
Copy quality

Em-dashes and AI filler patterns in the offer copy

Boris flagged: em-dashes are an AI tell.

Before any client-facing document leaves the draft stage, run the copy-editing sweep. Key checks: no em-dashes (use comma or full stop), no hedging language ("may", "could potentially"), no filler transitions ("furthermore", "it is worth noting"), active voice, sentences under 25 words. A single sweep catches 90% of AI-generated copy patterns that undermine trust.

9
Copy quality

Tool names appeared in the client document

Client-facing docs should not name Apify, PRAW, pytrends, BERTopic, etc.

Technical tool names belong in architecture docs for internal use. In client proposals and briefs, describe capability, not implementation. "We monitor 12 signal sources including Pinterest board saves and Reddit purchase discussions" - not "we use Apify Pinterest actor and PRAW". The client buys outcomes, not your stack.

10
PDF output

Physical address appeared in the PDF

Removed after design review.

PDFs are shared and can be cached publicly. Company address, VAT number, and registration details belong on invoices, not on commercial proposals. Keep proposals free of details that become liabilities if the document circulates beyond the intended recipient.

What Worked

Patterns that produced good output on the first try

A
Architecture

Separating intent signals from brand mention monitoring

The core thesis - that Meltwater/Brandwatch count PR volume while the client needs buyer intent - was right from the start and never had to be revisited. Frame the problem correctly before touching code. Write the "what this is not" section before the "what this is" section.

B
Architecture

Three-tier source model (intent / aesthetic / global early-warning)

Organising 12 sources into tiers by signal type - not by platform category - made the architecture immediately legible to the client. The client can evaluate each tier's relevance to their business without needing to understand the technical implementation.

C
Design briefs

Adding a design-brief output layer

The "design brief" section of the weekly report - translating signal data directly into product development direction - was proposed upfront and the client never questioned it. It is the clearest differentiator from commodity tools. Any intelligence system for a product company should include this layer.

D
Visuals

Publishing the system overview as an interactive HTML page

Instead of a static architecture diagram, we built an interactive HTML with phase filters, a comparison table, and a processing logic walkthrough. This gave the client something to share with their team that explains the system without requiring a live demo. Always ask: can this artefact sell itself without you in the room?

E
China layer

Justifying Xiaohongshu with quantified business rationale

When the client challenged the China data sources, having a ready response - "China = 30-40% of global luxury jewelry revenue; RED precedes Western aesthetic trends by 6-12 months; you can disable it if Asia is not a priority market" - turned a potential objection into a kept feature. Always have a one-paragraph "why this matters for you specifically" for each non-obvious source.

Recommended Process

The sequence that gets you to final quality in one pass

Step 1

Client profile brief (30 min)

Before any design: confirm ASP range, HQ country, billing currency, markets served, excluded geographies, competitor peer set (names to include and exclude), internal teams who receive the output, and whether any data sources are politically sensitive. A single Notion page with these answers prevents 6 of the 10 rework items above.

Step 2

Source validation pass

For each proposed source, write one sentence: "This source tells us [what buyer behaviour], which matters because [business decision it informs]." If you cannot write that sentence, cut the source. Present the shortlist to the client for confirmation before building any connectors.

Step 3

Commercial offer - first draft checklist

Before sending any draft: (1) currency matches client billing jurisdiction, (2) timeline includes 50% buffer on top of technical estimate, (3) costs split into three buckets (infrastructure / AAS work / upgrade reserve), (4) no tool names, no physical address, (5) copy-editing sweep run, (6) em-dashes removed, (7) VAT note included.

Step 4

Architecture doc - make it self-selling

Build the system overview as something the client can share internally without you. Interactive is better than static. Include: what the system is not (the competitive comparison), a phase-by-phase roadmap, and the processing logic in plain language. Publish to a permanent URL, not as a PDF attachment.

Step 5

Phase 1 proof run before automation

Run one real week manually. Show the client a complete output (PDF brief + data sources) before committing to full automation. This validates the signal quality and report format before you build the cron layer. One manual run is worth more than three weeks of automated output the client never looks at.

The meta-lesson Most rework in this build came from assumptions, not from technical mistakes. The code was right. The missing ingredient was a 30-minute client brief at the start that would have answered: segment, geography, currency, competitor set, politically sensitive inclusions. Run that brief before writing a single line.
